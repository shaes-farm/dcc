# ADR-0002: Provider methods return domain types, not §2.2's provisional shapes

**Date:** 2026-07-23
**Status:** accepted
**Spec:** §2.2, §3.0, §3.1
**Issue:** [#5](https://github.com/shaes-farm/dcc/issues/5)

## Context

§2.2 sketches the provider interfaces in TypeScript, and its method signatures
name return types that do not exist as canonical objects: `EnvSummary`,
`EnvDetail`, `Deploy`, `WorkloadStatus`, `ArtifactRef`, `DocumentRef`,
`DashboardRef`, `NormalizedSpec`. §3.1 then defines the canonical objects those
shapes describe — `Environment`, `Deployment`, `Workload`, `Artifact`,
`Document`, `Dashboard`, `Api` — and `lib/domain` implements them, each carrying
its normalized `status` and its URI.

So the same concept has two names: the provider signature (`Deploy`) and the
domain object (`Deployment`). Left unresolved, #5 would either mint duplicate
provider-layer types that shadow the domain ones — the "widening the model to
fit a provider" failure mode the architecture is organized to prevent — or leave
the signatures pointing at types that were never defined.

`lib/domain` already anticipated half of this. Its README and the header
comments in `observability.ts` state the split explicitly: the domain owns the
_addressable nodes_, and #5 owns the _query/filter/result payloads_
(`MetricQuery`, `Series`, `LogPage`). And the README's "`Provider` vs
`ProviderAdapter`" note already reconciled the one name §2.2 and §3.1 genuinely
collide on. What remained was the return-type names.

## Decision

Provider methods return **`lib/domain` types**. §2.2's provisional names map
onto the canonical objects one-to-one:

| §2.2 signature name | Returned domain type                                   |
| ------------------- | ------------------------------------------------------ |
| `EnvSummary`        | `Environment`                                          |
| `EnvDetail`         | `EnvironmentDetail` (new composite — see below)        |
| `Deploy`            | `Deployment`                                           |
| `WorkloadStatus`    | `Workload` (which already carries normalized `status`) |
| `ArtifactRef`       | `Artifact`                                             |
| `DocumentRef`       | `Document`                                             |
| `DashboardRef`      | `Dashboard`                                            |
| `NormalizedSpec`    | `Api` (with its `Operation`s)                          |

`lib/providers` defines only what has no domain equivalent: the
filter/query/result payloads (`PrFilter`, `RunFilter`, `LogQuery`, `LogPage`,
`MetricQuery`, `Series`, `LogSearch`, `SpecSource`, `PlaygroundRequest/Response`,
`DeployOptions`) and the base contract (`ProviderAdapter`, `Capability`,
`ConnectionResult`, `ActionResult`, `Scope`).

One genuine composite is added: **`EnvironmentDetail`** = `{ environment:
Environment; workloads: Workload[] }`. §2.2's `EnvDetail` ("workloads/pods") has
no single domain object because pods hang off workloads, not off the
environment; `getEnvironment` returns this pairing while `listEnvironments`
returns the bare `Environment` rollup.

The `Provider` → `ProviderAdapter` rename is **not** re-decided here; it is
recorded in `lib/domain/README.md` and merely cited.

`docs/ARCHITECTURE.md` is **not** edited — it stays frozen at v0.4. §2.2 is read
together with this ADR.

## Consequences

- There is exactly one shape per concept. A `Deployment` fetched by a panel and
  a `Deployment` returned by `getRecentDeploys` are the same type, so lineage
  matching, favorites, and graph edges compose without conversion.
- Reading §2.2 now requires this mapping table. That is the standing cost of a
  frozen spec, and why [index.md](index.md) exists.
- `EnvironmentDetail` is the one new node-ish type in the provider layer. It is a
  composition of domain types, not a new canonical object, so it needs no URI
  scheme and does not touch §3.2.
- The domain stays free of provider-shaped fields, keeping the
  `no-restricted-imports` boundary meaningful: normalization happens in
  `lib/providers`, and only domain types cross out of it.

## Alternatives considered

**Mirror §2.2's names literally.** Define `EnvSummary`, `Deploy`,
`WorkloadStatus`, `ArtifactRef`, etc. as provider-layer types exactly as
written. This duplicates the domain objects under second names, so every route
handler and panel would convert between `Deploy` and `Deployment`, and the two
would drift — the precise drift the single-vocabulary rule (§3.1) forbids. The
provisional names read as pseudocode shorthand, not a second type system.

**Push the composites into `lib/domain`.** Add `EnvironmentDetail` (and any
future `*Detail`) as canonical objects. Rejected: they are query-result
compositions, not addressable things — no URI in §3.2, no palette entry in §5.4.
Putting them in the domain would blur "addressable node" with "method payload",
the very line `observability.ts` draws.

**Amend `docs/ARCHITECTURE.md`.** Rewrite §2.2's signatures to name the domain
types. Cheapest to read, but it reopens a spec frozen on purpose (§11 names the
spec outrunning the software as the top risk). Recording the reconciliation
beside the spec keeps the freeze real and every post-freeze change reviewable.
